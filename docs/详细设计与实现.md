# 1 系统的详细设计与实现

## 1.1 数据同步功能的详细设计与实现

### 1.1.1 功能描述

数据同步功能是整个系统的数据基础，负责从腾讯官方 KPL 赛事 API 自动拉取联赛、赛程、对局、选手、英雄、战队、装备等全维度数据，并持久化到 MySQL 数据库中。系统支持全量同步、增量同步、深度同步三种模式，并通过定时任务实现每日自动更新。

核心技术栈：Spring Boot 3.2.5 + MyBatis-Plus 3.5.6 + MySQL + Flyway 数据库版本管理 + Redis 缓存。外部数据源包括腾讯赛事 API（prod.comp.smoba.qq.com）、TGA 银行 API（tga-openapi.tga.qq.com）、腾讯新闻 API（apps.game.qq.com）以及 pvp.qq.com 静态数据。

### 1.1.2 接口设计

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/sync/latest` | 同步最新赛季（浅层，不含对局详情） |
| POST | `/api/sync/all` | 全量同步 2019 年至今所有赛季 |
| POST | `/api/sync/year?year=X` | 同步指定年份 |
| POST | `/api/sync/latest/incremental` | 增量同步（刷新赛程 + 榜单） |
| POST | `/api/sync/latest/deep-incremental?matchLimit=10` | 深度增量（回填对局详情 + 装备） |
| POST | `/api/sync/season?leagueId=X&deep=false` | 同步指定赛季 |
| POST | `/api/sync/leagues` | 仅同步联赛列表 |
| POST | `/api/sync/equip-info` | 从 pvp.qq.com 同步装备基础信息 |
| POST | `/api/sync/inscription-info` | 从 pvp.qq.com 同步铭文信息 |
| POST | `/api/sync/fix-equip-encoding` | 修复装备名称乱码 |
| POST | `/api/sync/reset-positions?leagueId=X` | 重置并重新同步分路位置数据 |

### 1.1.4 控制层设计与实现

`SyncController` 位于 `controller/SyncController.java`，提供 RESTful 接口。每个同步接口接收可选参数（leagueId、year、deep、matchLimit 等），调用 `DataSyncService` 执行同步逻辑，返回统一的 `ApiResponse` 响应。

关键实现：同步操作为异步长任务，通过 `SyncJob` 实体记录任务状态（RUNNING/SUCCESS/FAILED），支持任务查询和重试。

### 1.1.5 业务层设计与实现

`DataSyncService` 是核心业务类，包含约 800 行代码，实现完整的数据同步链路：

**全量同步流程**：`syncAll()` 遍历 2019 年至今的所有赛季，依次调用 `syncLeagues()` → `syncMatches(leagueId)` → `syncBattles(matchId)` → `syncBattleDetail(battleId, leagueId)`，最后聚合生成 `hero_stats`、`player_stats`、`team_stats` 三张统计表。

**增量同步流程**：`syncLatestIncremental()` 仅刷新最新赛季的赛程和榜单统计，耗时约 10 秒。`syncLatestDeepIncremental()` 额外回填最近 N 场比赛的对局详情和装备数据。

**阵营归一化处理**：KPL API 的阵营编号（1/2）在不同对局中可能翻转。`syncBattleDetail()` 通过比对比赛级别的队伍 ID 与对局级别的一致性，检测并修正翻转的阵营数据。

**分路位置校正**：系统维护一个约 150 个英雄 ID 到标准分路的静态映射表（`HERO_POSITION`）。当 API 返回 position=0 或同一阵营内出现重复分路时，执行 4 策略修复循环：(1) HERO_POSITION 去重；(2) 填补缺失分路；(3) 使用 positionDesc 文本匹配；(4) 强制重新分配。

**定时同步**：`KplSyncScheduler` 通过 `@Scheduled` 注解实现每日 3:00 增量同步、每日 3:10 深度同步。

### 1.1.6 持久层设计与实现

系统使用 12 张数据库表，通过 Flyway 管理版本（V1-V10）。核心表结构：

- `league`：联赛基础信息（leagueId, leagueName, leagueType, year, season, status）
- `match`：赛程信息（matchId, leagueId, camp1TeamName, camp2TeamName, bo, winCamp, matchStage, startTime）
- `battle`：对局信息（battleId, matchId, battleSeq, winCamp, gameDuration）
- `battle_player`：对局选手详情（约 30 个字段，含 KDA、伤害、经济、分路、召唤师技能、铭文等）
- `battle_player_equip`：选手装备信息（equipId, equipName, equipIcon, equipOrder）
- `hero_stats` / `player_stats` / `team_stats`：聚合统计表
- `equip_info` / `inscription_info`：装备和铭文基础信息
- `sync_job` / `sync_cursor`：同步任务管理和游标

MyBatis-Plus 提供 CRUD 自动映射，`BattlePlayerMapper` 和 `BattlePlayerEquipMapper` 包含自定义 SQL 用于复杂聚合查询。

### 1.1.7 相关配置类及工具类的设计与实现

- **`RestTemplateConfig`**：配置 RestTemplate（5s 连接 / 10s 读取超时），对 `*.qq.com` 域名绕过代理，强制 UTF-8 编码
- **`MybatisPlusConfig`**：实现 `MetaObjectHandler`，自动填充 `createdAt`（插入时）和 `updatedAt`（更新时）
- **`RedisConfig`**：自定义 `RedisTemplate<String, Object>`，使用 Jackson JSON 序列化器（含 JavaTimeModule）
- **`SchedulerConfig`**：定时任务配置，每日 3:00 执行增量同步
- **`DotenvConfig`**：`ApplicationContextInitializer`，在 Spring 上下文启动前加载 `.env` 文件
- **`QueryCacheService`**：Redis 缓存封装，提供 `getOrLoad(key, ttl, loader)` 和 `evictByPattern(pattern)` 方法，缓存 TTL 600 秒

## 1.2 AI 智能分析功能的详细设计与实现

### 1.2.1 功能描述

AI 智能分析功能是系统的核心特色，用户通过自然语言提问（如"AG超玩会最近表现怎么样"、"ban率最高的英雄是谁"），系统自动识别意图、查询数据、调用 LLM 生成专业分析报告。系统采用三层架构：意图识别 → 工具路由 → 报告生成。

技术实现：LLM 使用小米 MiMo v2.5 推理模型（OpenAI 兼容协议），支持 SSE 流式输出，推理过程（chain-of-thought）和最终回答分开展示。当本地数据不足时，自动回退到 DuckDuckGo 网络搜索。

### 1.2.2 接口设计

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/agent/chat` | 同步对话，返回 `{"reply": "..."}` |
| POST | `/api/agent/chat/stream` | SSE 流式对话，推送 data/reasoning/content 三种事件 |

SSE 事件格式：
- `{"t":"data","d":[...]}` — 结构化查询结果
- `{"t":"r","d":"..."}` — 推理过程片段
- `{"t":"c","d":"..."}` — 最终回答片段
- `[DONE]` — 流结束标记

### 1.2.3 功能时序图

```
用户                AgentController       KplAgentService      AgentIntentRecognizer    Tool层           ReportGenerator        LLM(MiMo)
 |                      |                      |                      |                  |                    |                    |
 |-- POST /agent/chat ->|                      |                      |                  |                    |                    |
 |                      |-- chatStream() ------>|                      |                  |                    |                    |
 |                      |                      |-- recognize() ------->|                  |                    |                    |
 |                      |                      |                      |-- LLM意图分类 --->|                    |                    |
 |                      |                      |                      |<-- Intent -------|                    |                    |
 |                      |                      |<-- Intent ------------|                  |                    |                    |
 |                      |                      |-- routeToTool() ----->|                  |                    |                    |
 |                      |                      |                      |                  |-- 查询数据库 ------>|                    |
 |                      |                      |                      |                  |<-- 结构化数据 ------|                    |
 |                      |                      |<-- 数据 ---------------|                  |                    |                    |
 |                      |                      |-- push SSE data event |                  |                    |                    |
 |<-- SSE: data --------|                      |                      |                  |                    |                    |
 |                      |                      |-- generateStream() --|                  |                    |                    |
 |                      |                      |                      |                  |                    |-- 调用LLM --------->|
 |                      |                      |                      |                  |                    |<-- reasoning token -|

 |<-- SSE: r (推理) ----|                      |                      |                  |                    |                    |

 |<-- SSE: c (回答) ----|                      |                      |                  |                    |                    |
 |<-- [DONE] -----------|                      |                      |                  |                    |                    |
```

### 1.2.4 控制层设计与实现

`AgentController` 位于 `controller/AgentController.java`，提供两个端点：

- `/api/agent/chat`：同步模式，接收 `{"message":"..."}` JSON，调用 `KplAgentService.chat()` 返回完整回复
- `/api/agent/chat/stream`：SSE 流式模式，设置 `produces = MediaType.TEXT_EVENT_STREAM_VALUE`，调用 `KplAgentService.chatStream()` 返回 `Flux<String>`

### 1.2.5 业务层设计与实现

**意图识别层（AgentIntentRecognizer）**：使用 MiMo LLM 进行结构化意图分类，将用户问题映射为 9 种意图类型（PLAYER_QUERY、HERO_QUERY、TEAM_QUERY、MATCH_QUERY、HERO_TOP、TEAM_RANKING、TEAM_HONORS、EQUIP_QUERY、UNKNOWN），同时提取实体名称、分路、排序方式等结构化字段。当 LLM 返回 UNKNOWN 或调用失败时，回退到基于关键词的规则匹配。

**工具路由层（KplAgentService）**：根据意图类型路由到对应的 Tool 类，组合多个数据源提供丰富上下文。例如选手查询会同时返回基础数据、英雄池和常用装备。当数据为空或意图未知时，触发 `WebSearchService` 进行 DuckDuckGo 网络搜索回退。

5 个 Tool 类分别处理：选手统计（PlayerStatsTool）、英雄统计（HeroStatsTool）、战队统计（TeamStatsTool）、比赛分析（MatchAnalysisTool）、装备统计（EquipStatsTool）。其中装备查询支持 40+ 昵称映射（如"黑切"→"暗影战斧"），选手查询支持别名系统（如"AG一诺"→"一诺"）。

**报告生成层（ReportGenerator）**：将结构化数据组装为 prompt，调用 LLM 生成专业分析报告。System prompt 指导 LLM 扮演 KPL 专业数据分析师。支持同步（`generate()`）和流式（`generateStream()`）两种模式。流式模式使用 `java.net.http.HttpClient` 读取 SSE 行，分别处理推理内容（`reasoning_content`）和最终回答。当 LLM 不可用时，回退为格式化纯文本报告。

### 1.2.6 持久层设计与实现

AI 功能本身不直接操作数据库，而是通过 Tool 层调用 Mapper 查询已同步的数据。关键查询包括：

- `BattlePlayerMapper.aggregatePlayerStats()`：当 player_stats 表缺少数据时，从 battle_player 表实时聚合
- `BattlePlayerMapper.playerHeroStats()`：查询选手的英雄胜率数据
- `BattlePlayerMapper.heroPlayerStats()`：查询英雄的高胜率选手
- `BattlePlayerEquipMapper.countByPosition()`：装备的分路使用分布

### 1.2.7 相关配置类及工具类的设计与实现

- **`AiConfig`**：`@ConfigurationProperties(prefix="ai")`，持有 LLM API Key、Base URL、模型名称、网络搜索配置（开关、最大结果数、超时、查询前缀"KPL 王者荣耀职业联赛"）
- **`WebSearchService`**：DuckDuckGo HTML 抓取实现，无需 API Key，解析搜索结果的标题、摘要和链接
- **`KplAgentService.PLAYER_ALIASES`**：选手别名映射表，支持常见昵称自动匹配
- **`EquipStatsTool.EQUIP_NICKNAMES`**：40+ 装备昵称映射表

## 1.4 BP 模拟器功能的详细设计与实现

### 1.4.1 功能描述

BP 模拟器是一个模拟 KPL 赛事 Ban/Pick 流程的交互式工具，支持 BO3/5/7/9 多种赛制，完整还原官方 BP 规则（5 阶段禁用、双选机制、巅峰对决）。功能包括：多局系列赛管理、败方选边机制、30 秒倒计时、英雄分级选择器、阵容评分对比、撤销/重置操作等。

技术实现：纯前端 Vue 3 Composition API 实现，英雄数据通过 API 从后端获取并合并本地 HERO_CATALOG 静态数据。

### 1.4.2 接口设计

|------|------|------|
| GET | `/api/query/hero/top?sort=pick&limit=200` | 获取英雄列表及 BP 统计数据 |

### 1.4.4 控制层设计与实现

BP 模拟器为纯前端模块，无后端控制层。所有状态通过 Vue 3 的 `ref` 和 `computed` 响应式管理。

### 1.4.5 业务层设计与实现

核心逻辑在 `BpSimulatorPage.vue` 的 `<script setup>` 中实现：

**BP 序列生成**（`createBpSequence(firstPickSide)`）：根据先手方动态生成 20 步标准 BP 序列（10 禁 + 10 选）。第一阶段：双方各 Ban 2；选 3 轮英雄（1-2-2 分配）；第二阶段：双方各 Ban 3；选 2 轮英雄（1-2-1 分配）。巅峰对决使用独立的 10 步盲选序列。

**游戏状态管理**（`createGameState()`）：每局包含 blueBans[5]、redBans[5]、bluePicks[5]、redPicks[5]、currentStep、firstPickSide、winner、blindPick 等字段。

**败方选边机制**：标记胜负后自动弹出选边对话框。普通局由败方选边；巅峰对决由先拿到赛点的队伍优先选边（通过 `firstToMatchPoint()` 函数计算）。3:3 平局时不弹窗，败方默认坐蓝方。

**英雄选择器**：支持名称搜索、分路筛选、梯度筛选（T0-T4），英雄按梯度分组展示。梯度基于 `heroScore = (pickRate + banRate) * 50 + banRate * 30 + winRate * 20` 公式动态计算。

**阵容评分**：蓝红双方已选英雄的评分行对比，含单英雄评分和总分。

### 1.4.6 持久层设计与实现

BP 模拟器无持久层，所有状态保存在 Vue 响应式变量中，页面刷新后重置。

### 1.4.7 相关配置类及工具类的设计与实现

- **`heroData.js`**：`HERO_CATALOG` 静态英雄数据（142 个英雄，含 heroId、heroName、role），作为 API 数据的补充和回退
- **英雄海报焦点映射**（`heroPosterFocus`）：为部分英雄自定义海报图片的裁切焦点位置，优化竖版卡片的视觉效果

## 1.5 对位雷达图功能的详细设计与实现

### 1.5.1 功能描述

对位雷达图是一个用于对比同位置两名选手在单局或系列赛中表现的可视化工具。系统通过百分位排名算法，将选手的 8 项核心指标（伤害占比、分均伤害、生存能力、参团率、KDA、经济效率等）映射到雷达图上，直观展示双方的优劣势。功能支持联赛/比赛/对局三级选择器、5 分路切换、单局/系列赛汇总模式、滚轮/触摸滑动切换分路。

### 1.5.2 接口设计

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/query/radar/lane?leagueId=X&matchId=X&battleId=X&role=X` | 获取对位雷达数据 |

返回数据包含：双方选手基础信息、8 维雷达评分、亮点数据（总伤害、击杀、参团率等）、装备配置、召唤师技能、铭文信息。

### 1.5.3 功能时序图

```
用户              LaneRadarPage.vue        QueryController       LaneRadarService        Mapper
 |                     |                        |                      |                    |
 |--- 选择联赛/比赛 --->|                        |                      |                    |
 |--- 选择分路 -------->|                        |                      |                    |
 |                     |--- fetch /radar/lane ->|                      |                    |
 |                     |                        |--- buildRadar() ---->|                    |
 |                     |                        |                      |--- 查询双方选手数据 ->|
 |                     |                        |                      |--- 构建样本池 ------>|（当前联赛/近2赛季/全历史）
 |                     |                        |                      |--- 计算百分位排名 -->|
 |                     |                        |                      |--- 生成雷达评分 ---->|
 |                     |                        |<-- RadarVO -----------|                    |
 |                     |<-- JSON响应 ------------|                      |                    |
 |--- SVG雷达图渲染 --->|                        |                      |                    |
 |--- 英雄面板渲染 ---->|                        |                      |                    |
```

### 1.5.4 控制层设计与实现

`QueryController` 中的 `/api/query/radar/lane` 端点，接收 `leagueId`、`matchId`、`battleId`、`role` 四个参数，调用 `LaneRadarService.buildRadar()` 返回雷达数据。

### 1.5.5 业务层设计与实现

**后端 LaneRadarService**：核心算法为百分位排名。首先构建样本池（优先当前联赛，不足时扩展到近 2 个赛季或全历史），然后对每个指标计算当前选手在样本中的百分位排名（0-100），最终映射为 8 维雷达评分。支持单局模式（指定 battleId）和系列赛汇总模式（指定 matchId，聚合所有对局数据）。

**前端 LaneRadarPage.vue**：使用 SVG 绘制雷达图（8 条轴线、多边形区域、网格线），两侧英雄面板展示大图背景、选手头像、KDA、经济、装备、召唤师技能、铭文。高亮卡片展示关键对比数据。支持鼠标滚轮和触摸滑动切换分路，移动端强制横屏显示。

### 1.5.6 持久层设计与实现

雷达数据为实时计算，不缓存。依赖 `battle_player` 表的历史数据构建样本池。

### 1.5.7 相关配置类及工具类的设计与实现

- **`heroData.js`**（前端）：英雄静态数据，用于 API 数据缺失时的回退
- **海报图片淡入动画**：CSS `opacity: 0` + `transition: opacity .15s ease` + `@load` 事件设置 `opacity: 1`，实现图片加载完成后的无感淡入

## 1.6 英雄梯度榜与装备分析功能的详细设计与实现

### 1.6.1 功能描述

英雄梯度榜和装备分析是两个独立但结构相似的功能模块，均采用百分位分档算法将数据分为 T0-T4 五个梯度。英雄梯度榜基于 BP 率、禁用率、胜率的加权评分公式；装备分析基于分路出场次数或整体出场率。两个模块均支持分路筛选、详情弹窗、亮/暗主题切换。

### 1.6.2 接口设计

| 方法 | 路径 | 说明 |

| GET | `/api/query/equip/detail?equipId=X` | 装备详情 |
| GET | `/api/query/equip/hero?heroName=X` | 英雄常用装备 |

 |                     |                      |                      |

通过 `QueryController` 的现有端点获取数据，无额外控制层。

### 1.6.5 业务层设计与实现

**英雄梯度榜（TierListPage.vue）**：评分公式 `score = BP率 * 50 + 禁用率 * 30 + 胜率 * 100`。英雄从 `heroRoleMap` 静态映射获取分路信息。百分位分档：T0（前 10%）、T0.5（10-25%）、T1（25-50%）、T2（50-75%）、T3（75-90%）、T4（后 10%）。高禁用率（>30%）英雄卡片标红边框，低出场（<10 场）标蓝色虚线边框。

**装备分析（EquipmentPage.vue）**：装备按分路出场次数评分（选择具体分路时）或整体出场率评分（选择全部时）。同样使用百分位分档算法。详情弹窗展示装备属性描述、被动效果、分路使用分布柱状图、热门英雄列表。

### 1.6.6 持久层设计与实现

通过 `BattlePlayerEquipMapper` 的自定义查询获取装备的分路分布和英雄使用数据。

- **`BUGGY_POS_FIX`**（EquipmentPage.vue）：历史数据分路映射修正表，处理 API 返回的异常分路数据

## 1.7 英雄画廊功能的详细设计与实现

### 1.7.1 功能描述

英雄画廊是一个展示王者荣耀全英雄原画和皮肤的海报式画廊。支持分路筛选、名称搜索、皮肤切换（仅显示基础皮肤/显示全部皮肤）、IntersectionObserver 懒加载、详情弹窗（全尺寸原画展示、皮肤选择器、皮肤元数据）。数据来自三个来源的合并：后端 API、腾讯 herolist.json、zlkdatasys JSON。

### 1.7.2 接口设计

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/query/hero/top?sort=pick&limit=200` | 英雄基础数据 |
| GET | `/skin-api/web201605/js/herolist.json` | 腾讯英雄/皮肤名称列表（代理） |
| GET | `/skin-api/zlkdatasys/data_zlk_xpflby.json` | 皮肤详情元数据（代理） |

### 1.7.3 功能时序图

```
用户              HeroGalleryPage.vue       后端API          腾讯外部API
 |                      |                      |                  |
 |--- 页面加载 -------->|                      |                  |
 |                      |--- fetch hero/top -->|                  |
 |                      |--- fetch herolist ---|----------------->|
 |                      |--- fetch zlkdata ----|----------------->|
 |                      |<-- 三个数据源 --------|                  |
 |                      |--- mergeHeroes() --->| 合并去重           |
 |                      |--- 渲染海报网格 ----->|                  |
 |                      |                      |                  |
 |--- 滚动到可视区 ----->|                      |                  |
 |                      |--- IntersectionObserver 触发 ---------->|
 |                      |--- 加载图片 --------->|                  |
 |                      |--- @load淡入动画 --->|                  |
 |                      |                      |                  |
 |--- 点击海报 -------->|                      |                  |
 |                      |--- 弹窗: 全尺寸原画 + 皮肤选择器 ------>|
```

### 1.7.4 控制层设计与实现

通过 Vite 代理将 `/skin-api/*` 请求转发到 `https://pvp.qq.com`，避免跨域问题。

### 1.7.5 业务层设计与实现

**数据合并**：三个数据源通过 heroId 去重合并，API 数据优先，外部数据补充皮肤名称、发布时间、海报 URL 等元数据。`MANUAL_SKIN_DATA`（skinData.js）手动补充缺失的皮肤数据。

**懒加载**：自定义 `v-lazy` 指令，基于 IntersectionObserver 监听元素进入可视区后设置 `img.src`。

**海报焦点调优**：`heroPosterFocus` 和 `rolePosterFocus` 为每个英雄/分路定义海报图片的裁切焦点（x, y, scale），优化竖版卡片的视觉效果。

**图片淡入**：所有图片初始 `opacity: 0`，`@load` 事件触发后设置 `opacity: 1`，CSS `transition: opacity .15s ease` 实现无感淡入。

### 1.7.6 持久层设计与实现

画廊数据为前端实时合并，不持久化。英雄基础数据来自后端 API 缓存。

### 1.7.7 相关配置类及工具类的设计与实现

- **`heroData.js`**：`HERO_CATALOG`（142 英雄），作为 API 数据缺失时的回退
- **`skinData.js`**：`MANUAL_SKIN_DATA`，手动维护的皮肤元数据
- **Vite 代理配置**：`/skin-api` → `https://pvp.qq.com`，自动重写路径
